1. お客さんの物件検索

使うサービス（4つ）

  • REINS: ログイン必須。各サイトで1番遅いが情報量◎
  • itandi (BB): ログイン必須。API直叩きで超速い
  • いえらぶ (BB): ログイン必須。URLパラメータで絞り込み
  • ES-Square: ログイン必須。広告可バッジでフィルタ

お客さん新規登録

  1. AdminPage（検索条件管理）を開く
  2. 「— 新規顧客 —」を選択した状態で、顧客名・LINE User ID（任意）を入力
  3. エリア（路線・駅 or 市区町村）を選択 ※同時併用不可
  4. 賃料・間取り・徒歩・面積・築年数・構造・設備を入力
  5. 「保存」

検索の起動

  • 手動: 拡張アイコン → ダッシュボード → 「検索開始」
  • 自動: 定刻で cron トリガー（実装次第）

結果通知

  • 新着物件 → Discord（forum チャンネルにスレッド形式）
  • LINE User ID 登録済みお客さん → LINE にも条件マッチ通知（実装次第）

賃料下限の挙動

  • 顧客の rent_min 指定あり → その値を下限
  • 顧客の rent_min 未指定 → rent_max の70% を自動で下限に設定（極端に安い物件を除外）
  • SUUMO巡回時は _isSuumoPatrol フラグで70%自動設定をスキップ
バス・トイレ別の処理は拡張オプションで「alert / skip」を切替可能。skip にすると条件不一致で自動除外、alert は通過してアラート表示のみ。

2. SUUMO巡回（自動入稿）

全体フロー

巡回条件で4サイト検索 → 新着発見
  ↓
SUUMO候補物件シートに保存（status=新着）
  ↓
Discord通知（thread形式） → 「承認ページを開く」リンク
  ↓
承認ページで内容確認 + 必須項目チェック + 画像ジャンル選択
  ↓
「承認して入稿」クリック → submitting
  ↓
ForRent タブで自動入稿（Phase1〜5）
  ↓
登録完了画面検知（Phase6） → posted + SUUMO掲載管理シートに行追加

巡回条件の管理

  1. AdminPage → 「SUUMO巡回条件を管理」
  2. 条件は最大50件枠で運用（active状態の上限）
  3. 各条件: 名前 / 路線・駅 or 市区町村 / 賃料 / 間取り / 徒歩 / 面積 / 築年数 / 構造 / 設備 / 巡回間隔
  4. 巡回間隔は「N日以内」（各サイトの新着絞り込み）

承認画面の必須項目

SUUMO 入稿で必須の項目には承認画面でマーク表示。承認ボタン押下時にバリデーション。

  • 🟥 ■ = SUUMO入稿で必須（空だとエラー）: 建物名 / 住所 / 賃料 / 所在階 / 総階数
  • 🟧 ▲ = SUUMO掲載必須: 構造 / 築年数 / 最寄駅 / 間取り / 面積 / 入居可能時期 / 元付会社 / 元付電話番号
  • 🟨 ⚠ = 条件付き必須: 定期借家のとき契約期間（期間または期限）

画像ジャンルとスコア

分類カテゴリ配点
主要間取り / 外観 / リビング / キッチン / バス各5点（max 25点）
その他トイレ・洗面・収納・その他部屋・玄関 等1ジャンル1点（max 9点）

合計34点満点で32点が目標。同じジャンル複数枚使っても加点は1回のみ。

ホームズ画像検索ボタンで補完候補を表示 → 採用予定込みの「予測スコア」が画面下部のフローティングバーに常時表示。

反響予測スコア（人気証跡）

巡回時にDiscord通知の冒頭に表示される 📊 反響予測: 74点 👍 標準以上 のスコア。 物件の 本来の人気度（=スペック上の魅力） を 0〜100 点で算出する。

📍 何を測っているか: 「相場と比べて割安か？」「駅から近いか？」「築年数は新しいか？」の 3 つの軸で物件のスペック魅力を点数化。 設備が多いほど期待値（相場）がブーストされ、設備込みで割安なら高評価。

軸1: 平米単価 vs 期待相場（重み60%）— 一番効く

同じエリア・間取り・築年数・徒歩のSUUMO相場中央値（円/㎡）を取得し、自分の物件の平米単価と比較。

期待月額 = 相場中央値(円/㎡) × 面積 + (設備数 × 5,000円)
期待平米単価 = 期待月額 ÷ 面積
実平米単価 = (賃料+管理費) ÷ 面積
偏差% = (期待 − 実) ÷ 期待 × 100   ← +なら割安、−なら割高
score1 = 50 + 偏差% × 1.67        ← 偏差0%なら50点、+30%で100点、−30%で0点

設備プレミアム（1つにつき +5,000円/月 相当）の対象設備:

  • オートロック / 宅配ボックス / 独立洗面 / 追い焚き / モニター付インターホン
  • ウォークインクローゼット / 浴室乾燥 / 2F以上 / 敷金なし / 礼金なし

例: 設備5個ある物件は相場+25,000円が「期待値」になる → その期待より安ければ「お得な物件」として高評価。

軸2: 駅徒歩（重み25%）

score2 = max(0, 100 − (徒歩分 − 1) × 5)
徒歩1分5分10分15分20分21分以上
score210080553050

軸3: 築年数（重み15%）

築15年以内: score3 = 100 − 築年数 × 3
築15年超:   score3 = max(20, 55 − (築年数 − 15) × 1.5)
築年数新築5年10年15年20年30年50年
score3100857055483320

築古はそこそこ減点されるが、築15年以降は緩やかになり、最低20点で止まる（割安+設備が揃えば築古でも反響は出るため）。

総合スコア

反響予測スコア = score1 × 0.60 + score2 × 0.25 + score3 × 0.15
合計ラベル意味
80以上🔥 高反響予測条件が揃った優良物件
60-79👍 標準以上反響が期待できる
40-59➖ 普通可も不可もなく
20-39🔻 反響少なめ条件が弱い
0-19⚠️ 厳しい反響期待しづらい

計算例

武蔵境駅徒歩7分・築12年・35㎡・賃料13万・管理費3,000円・設備5個 の場合:

実平米単価 = (130,000 + 3,000) ÷ 35 = 3,800円/㎡
相場中央値 ≒ 3,500円/㎡ とすると…
期待月額 = 3,500 × 35 + 5 × 5,000 = 147,500円
期待平米単価 = 147,500 ÷ 35 = 4,214円/㎡
偏差% = (4,214 − 3,800) ÷ 4,214 × 100 ≒ +9.8%(割安)
score1 = 50 + 9.8 × 1.67 ≒ 66点
score2 = 100 − (7 − 1) × 5 = 70点
score3 = 100 − 12 × 3 = 64点
総合  = 66×0.6 + 70×0.25 + 64×0.15 ≒ 67点 → 👍 標準以上

計算ロジック実体: chrome_extension/inquiry-score.js の calculateInquiryScore()

このスコアは候補物件シートと掲載管理シートにも保存され、停止候補判定で「本来人気だが競合に埋もれた物件」を保護するのに使う。

停止候補ロジック（入れ替え）

いつ走る？

SUUMO 掲載枠 50 件が埋まっている時、新規承認の前に findStopCandidates が呼ばれて「どれを落とすか」を選出する。

2段構えの判定

各物件は順番にこの2つを通る:

  1. Step 1 保護ルール: 一定条件を満たす物件は そもそも落とし候補に入れない（safe扱い）
  2. Step 2 危険度スコア: 残った物件に点数をつけて、高い順に落とす

Step 1: 保護ルール

下の条件のどれかに当てはまれば 絶対に落とされない（候補にすら入らない）。

relaxLevel が 0→3 と段階的に緩むのは「全員保護対象で詰んだ時の救済」用。通常運用は level 0。

⚠️ 強制落とし対象（保護スキップ）
以下のいずれかに該当する物件は relaxLevel に関わらず保護対象外。Step 2 の +9999 ボーナスで確実に落ちる:
  • シート掲載日数 ≥ 60 日（長く掲載しても結果出てない）
  • 問い合わせ数 ≥ 30 件 かつ 初回申込検知あり（反響十分かつ売れた = 役目を果たした）
📝 問い合わせ ≥ 30 + 申込検知なし の場合
反響30件超えていても 他サイト（物件空室管理シート）で申込検知されていない なら保護維持。「反響あるのに申込来てない＝何か魅力ある可能性」として落とさない。
level保護される物件（強制落とし対象外 が前提）
0（標準）掲載 < 7 日 or 問合せあり&掲載 < 45 日 or 反響予測 ≥ 70 点
1（緩い）掲載 < 3 日 or 問合せあり&掲載 < 30 日 or 反響予測 ≥ 80 点
2（最小）掲載 < 1 日 or 反響予測 ≥ 90 点
3（無視）誰も保護されない（最後の手段）

level 0 で候補が見つからない時だけ自動で level 1→2→3 と緩める。最初に1件以上見つかった level で確定。

Step 2: 危険度スコア

保護ルールを通過した物件にこの式で点数をつける。

方向性: スコアが 高い ほど「落とすべき物件」、低い ほど「残したい物件」。
危険度スコア = 競合の脅威 − 問合せの好材料 − 反響予測の好材料
            − 申込スピード補正 − 競合resistance補正
            + 長期掲載ボーナス + 反響30+申込検知ボーナス
📊 各要素の意味

① 競合の脅威（プラス要素＝落とす理由）

weightedComp = 第3基準値の競合 × 2.1
             + 第2基準値の競合 × 1.6
             + 第1基準値の競合 × 1.0

競合脅威 = weightedComp × 10
  • SUUMOビジネスの「競合物件数（第1/第2/第3基準値）」をそのまま使う
  • 第3基準値（一番近い競合）が一番重い × 2.1。離れた競合（第1）は × 1.0
  • 例: lv3=5件、lv2=10件、lv1=20件 → (5×2.1+10×1.6+20×1.0)×10 = 465

② 問合せの好材料（マイナス要素＝守る理由）

− 問合せ数 × 100
  • 1 問合せで −100 点（= 第1基準値競合 10 件分の好材料）
  • 「実際に客が動いてる物件」を強く守る
  • 例: 問合せ 3 件 → -300

③ 反響予測の好材料（マイナス要素＝守る理由）

− 反響予測スコア × 30
  • 反響予測 1 点で −30 点
  • 例: 反響予測 60 点 → -1800（第1基準値競合 180 件分の好材料）
  • ※ 70 点以上は Step 1 で完全保護されるので、Step 2 で見るのは 0〜69 点の物件のみ

④ 申込スピード補正（マイナス要素＝守る理由）

− 1500 × (1 − 申込までの日数 / 30)²    （30日以上 or 検知なし は 0）
  • 物件空室管理シート (PROPERTY_SHEET_ID) の K列「終了日」を参照
  • 建物名 + 部屋番号 で SUUMO掲載管理シート と突合
  • 初回検知日は SUUMO掲載管理シート 22列目「初回申込検知日」に永続記録 (キャンセル後も保持)
  • 掲載開始 → 初回検知 までが早いほど人気物件として強く保護
申込までの日数補正意味
0 日（即日）−1500超人気
3 日−1230人気
7 日−867やや人気
14 日−427普通
21 日−135遅め
30 日以上 / 検知なし0効果なし

⑤ 競合 resistance 補正（マイナス要素＝守る理由）

− min(1500, (一日詳細PV / weightedComp) × 200)    （weightedComp < 5 はスキップ）
  • 「競合多いのに 詳細PV ある = 埋もれずに見られている = 本来人気」を表現
  • 一日詳細PV = 合計詳細PV / 実効掲載日数
  • 競合がそこそこ多い物件（weightedComp ≥ 5）のみ評価対象
  • 上限 −1500（極端な値の暴走防止）
競合 lv3weightedComp一日PVresistance補正
10件2150.24−48
10件21502.38−476
30件63200.32−63
30件631001.59−317
100件2105002.38−476

⑥ 長期掲載ボーナス（プラス要素＝諦めて落とす）

+ (シート掲載日数 ≥ 60 日 ? +9999 : 0)
+ (実効掲載日数 ≥ 45 日 ? +500 : 0)
  • 60 日超: +9999（実質的に問答無用で最優先落とし）
  • 45 日超: +500（強い落とし圧）
  • 「長く掲載しても結果出ない物件」を回転させるための強制力

⑦ 反響30+申込検知ボーナス（プラス要素＝役目果たしたら落とす）

+ (問い合わせ数 ≥ 30 && 初回申込検知あり ? +9999 : 0)
  • 反響が十分溜まって、実際に申込も検知されている = 物件としての役目を果たした
  • +9999 で最優先落とし
  • ※ 申込検知なし (V列 空) の場合は発動しない → 保護維持

物件競合lv3問合せ反響予測掲載日数計算スコア運命

狙い: パターン3（後から見られる物件）の救済
新規掲載後すぐに競合が増えるけど、競合落ち後に反響が出始める物件を早期に殺さないため、 反響予測スコア（スペックの魅力度）を強力なマイナス補正にしている。スペックが良い物件は 「今は競合多い」「問合せまだ来ない」状態でも、Step 1 で完全保護 or Step 2 で大幅減点される。

📊 評価機能（Phase A）

停止候補ロジックがちゃんと機能しているか後から検証できるように、以下のデータを蓄積する仕組みを入れた:

  • SUUMO 掲載管理シート 23列目「スコア内訳」: 各物件の最新スコア計算結果を JSON で保持。 どの補正がどれだけ効いているか一目でわかる

📈 競合トレンド検知

「SUUMO競合履歴」シートに日次の競合データをマトリクス形式（PV履歴と同じ構造）で記録。

  • 固定列: 物件名 / 部屋番号 / SUUMOコード / 種別(競合_第1/第2/第3)
  • 日付列: 横に並ぶ（90日保持）

effectiveDays = max(sheetDays, suumoListedDays)。シート日数とSUUMO日数の大きい方を使う。 SUUMO側は45で頭打ちなので、それより長く掲載されている物件はシート日数を使う。

submitting 放置物件の救済

承認後に自動入稿が失敗してユーザーが手動完結した場合の救済UIがAdminPageにある。

  • AdminPage の上部に「⚠️ SUUMO入稿で submitting のまま放置されている物件」セクション（0件なら非表示）
  • 「手動で入稿完了済みにする」 → posted化 + SUUMO掲載管理シートに行追加
  • 「却下扱いで戻す」 → submitting解除 → approved に戻る（再承認可能）
Phase6 の登録完了画面自動検知が動けば、放置されることはほぼ無くなる。AdminPage の救済UIは保険。

3. ログの読み方

確認場所

  • 拡張アイコン → 「ダッシュボードを開く」 → log.html
  • chrome://extensions/ → 「サービスワーカーを検査」 → Console

検索結果のスキップ集約ログ

各サイトの顧客検索終了時に「内訳」が1行で出る:

[いえらぶ] こが: スキップ内訳 → 通知済み 6件 / フィルタ済キャッシュ 10件
  • 通知済み: 過去にこの顧客に通知済みの物件（room_id一致）
  • フィルタ済キャッシュ: 前回フィルタで除外した物件を再スキップ（ielove のみ実装）

個別ログ（理由付き）が出るもの:

  • 申込あり（一覧/詳細段階）
  • 条件不一致（南向きでない、徒歩超過、etc）
  • 元付除外キーワード（SUUMO巡回時）
  • SUUMO競合多数（SUUMO巡回時）
  • 他サイトで申込あり（前回実行）

REINS画像アップロードログ

顧客名: REINS画像 base64取得=10件
顧客名: REINS画像アップロード完了 10/10件 via=catbox:8,0x0:2
  • via=... でどの画像ホスティングサービスで成功したか分かる
  • 失敗時は (失敗:N) [b64size=... err=...] がつく
  • ホストは catbox → 0x0 → telegraph → imgbb → freeimage → tmpfiles → pixeldrain → imgur の順でフォールバック
  • 失敗ホストは30分クールダウン（chrome.storage に永続化）

4. トラブル時の対処

検索が途中で止まる

  1. まずダッシュボードで最後のログを確認（どのサイト・どの段階で止まった？）
  2. 各サイトのログイン切れ → 該当サイトに手動ログイン → 再検索
  3. Service Worker が固まってる → chrome://extensions/ → サービスワーカーを「停止」 → 自動で再起動
  4. それでもダメ → 拡張アイコン右クリック → 「拡張機能を管理」 → 更新ボタン（🔄）で再読み込み

REINS画像アップロードが遅い・止まる（47秒以上）

  • ログの via=... を確認 → どのホストが効いているか
  • catbox長期障害 → 自動で30分クールダウンしてフォールバック
  • 2週間に1回・朝起こる謎の遅延 → SW Cold Start + DNS遅延の可能性。1回成功すれば治る
  • 全ホストダメ → all_hosts_failed: ... がログに出る

SUUMO入稿が失敗した

  • 確認画面でエラー出た → 手動でエラー修正して登録ボタン押下
  • 登録完了画面に到達すれば Phase6が自動検知 → 60分以内ならGASに通知される
  • 60分超えてしまった or 一度諦めた → AdminPage の「submitting 放置物件」セクションで救済

SUUMO候補物件が submittingで残ってる

  • AdminPage 上部に表示される
  • 「手動で入稿完了済みにする」or「却下扱いで戻す」で対処
  • 30分タイムアウトで自動的に approved に戻る仕組みもある

Chrome拡張の変更が反映されない

  • chrome_extension はローカル読み込み → chrome://extensions/ で更新ボタン押下
  • Service Worker は5分idle で終了 → 次のイベントで新コードがロードされる

GASコードの変更が反映されない

  • gas_deploy.sh 実行で全 deployment を新版に更新
  • nice-feistel ワークツリーが main と同期されている必要あり（gas_deploy.shが警告してくれる）

1. お客さんの物件検索

使うサービス（4つ）

  • REINS: ログイン必須。各サイトで1番遅いが情報量◎
  • itandi (BB): ログイン必須。API直叩きで超速い
  • いえらぶ (BB): ログイン必須。URLパラメータで絞り込み
  • ES-Square: ログイン必須。広告可バッジでフィルタ

お客さん新規登録

  1. AdminPage（検索条件管理）を開く
  2. 「— 新規顧客 —」を選択した状態で、顧客名・LINE User ID（任意）を入力
  3. エリア（路線・駅 or 市区町村）を選択 ※同時併用不可
  4. 賃料・間取り・徒歩・面積・築年数・構造・設備を入力
  5. 「保存」

検索の起動

  • 手動: 拡張アイコン → ダッシュボード → 「検索開始」
  • 自動: 定刻で cron トリガー（実装次第）

結果通知

  • 新着物件 → Discord（forum チャンネルにスレッド形式）
  • LINE User ID 登録済みお客さん → LINE にも条件マッチ通知（実装次第）

賃料下限の挙動

  • 顧客の rent_min 指定あり → その値を下限
  • 顧客の rent_min 未指定 → rent_max の70% を自動で下限に設定（極端に安い物件を除外）
  • SUUMO巡回時は _isSuumoPatrol フラグで70%自動設定をスキップ
バス・トイレ別の処理は拡張オプションで「alert / skip」を切替可能。skip にすると条件不一致で自動除外、alert は通過してアラート表示のみ。

2. SUUMO巡回（自動入稿）

全体フロー

巡回条件で4サイト検索 → 新着発見
  ↓
SUUMO候補物件シートに保存（status=新着）
  ↓
Discord通知（thread形式） → 「承認ページを開く」リンク
  ↓
承認ページで内容確認 + 必須項目チェック + 画像ジャンル選択
  ↓
「承認して入稿」クリック → submitting
  ↓
ForRent タブで自動入稿（Phase1〜5）
  ↓
登録完了画面検知（Phase6） → posted + SUUMO掲載管理シートに行追加

巡回条件の管理

  1. AdminPage → 「SUUMO巡回条件を管理」
  2. 条件は最大50件枠で運用（active状態の上限）
  3. 各条件: 名前 / 路線・駅 or 市区町村 / 賃料 / 間取り / 徒歩 / 面積 / 築年数 / 構造 / 設備 / 巡回間隔
  4. 巡回間隔は「N日以内」（各サイトの新着絞り込み）

承認画面の必須項目

SUUMO 入稿で必須の項目には承認画面でマーク表示。承認ボタン押下時にバリデーション。

  • 🟥 ■ = SUUMO入稿で必須（空だとエラー）: 建物名 / 住所 / 賃料 / 所在階 / 総階数
  • 🟧 ▲ = SUUMO掲載必須: 構造 / 築年数 / 最寄駅 / 間取り / 面積 / 入居可能時期 / 元付会社 / 元付電話番号
  • 🟨 ⚠ = 条件付き必須: 定期借家のとき契約期間（期間または期限）

画像ジャンルとスコア

分類カテゴリ配点
主要間取り / 外観 / リビング / キッチン / バス各5点（max 25点）
その他トイレ・洗面・収納・その他部屋・玄関 等1ジャンル1点（max 9点）

合計34点満点で32点が目標。同じジャンル複数枚使っても加点は1回のみ。

ホームズ画像検索ボタンで補完候補を表示 → 採用予定込みの「予測スコア」が画面下部のフローティングバーに常時表示。

入稿フローの段階

Phase場所処理
1承認画面imgbb等に画像アップロード、property_data_json構築
2ForRent入力フォーム各フィールドに自動入力 → 「確認画面へ」クリック
3確認画面必須項目チェック → 画像数チェック
4確認画面#jikko（登録ボタン）クリック
5遷移先URL変化を確認 → 成功なら GAS に suumo_post_complete(success:true)
6登録完了画面「登録完了しました」DOM検知 → 手動完結ケースを自動でGAS通知

停止候補ロジック（入れ替え）

50件枠が埋まっている時、新規承認の前に findStopCandidates が呼ばれて落とす候補を選出。

保護ルール（候補対象外、relaxLevel 0→3で段階的に緩和）:

  • level 0: 新着7日 / 問合あり<45日
  • level 1: 新着3日 / 問合あり<30日
  • level 2: 新着1日のみ
  • level 3: 保護なし

危険度スコア:

weightedComp = compLv3 × 2.1 + compLv2 × 1.6 + compLv1 × 1.0
riskScore = weightedComp × 10
         − inquiries × 100      (1問合せ = 競合10件相殺)
         + (sheetDays ≥ 60 ? 9999 : 0)
         + (effectiveDays ≥ 45 ? 500 : 0)

スコア降順で上位を停止候補に。SUUMOビジネスの第1〜3基準値競合数を使う。

submitting 放置物件の救済

承認後に自動入稿が失敗してユーザーが手動完結した場合の救済UIがAdminPageにある。

  • AdminPage の上部に「⚠️ SUUMO入稿で submitting のまま放置されている物件」セクション（0件なら非表示）
  • 「手動で入稿完了済みにする」 → posted化 + SUUMO掲載管理シートに行追加
  • 「却下扱いで戻す」 → submitting解除 → approved に戻る（再承認可能）
Phase6 の登録完了画面自動検知が動けば、放置されることはほぼ無くなる。AdminPage の救済UIは保険。

3. ログの読み方

確認場所

  • 拡張アイコン → 「ダッシュボードを開く」 → log.html
  • chrome://extensions/ → 「サービスワーカーを検査」 → Console

検索結果のスキップ集約ログ

各サイトの顧客検索終了時に「内訳」が1行で出る:

[いえらぶ] こが: スキップ内訳 → 通知済み 6件 / フィルタ済キャッシュ 10件
  • 通知済み: 過去にこの顧客に通知済みの物件（room_id一致）
  • フィルタ済キャッシュ: 前回フィルタで除外した物件を再スキップ（ielove のみ実装）

個別ログ（理由付き）が出るもの:

  • 申込あり（一覧/詳細段階）
  • 条件不一致（南向きでない、徒歩超過、etc）
  • 元付除外キーワード（SUUMO巡回時）
  • SUUMO競合多数（SUUMO巡回時）
  • 他サイトで申込あり（前回実行）

REINS画像アップロードログ

顧客名: REINS画像 base64取得=10件
顧客名: REINS画像アップロード完了 10/10件 via=catbox:8,0x0:2
  • via=... でどの画像ホスティングサービスで成功したか分かる
  • 失敗時は (失敗:N) [b64size=... err=...] がつく
  • ホストは catbox → 0x0 → telegraph → imgbb → freeimage → tmpfiles → pixeldrain → imgur の順でフォールバック
  • 失敗ホストは30分クールダウン（chrome.storage に永続化）

4. トラブル時の対処

検索が途中で止まる

  1. まずダッシュボードで最後のログを確認（どのサイト・どの段階で止まった？）
  2. 各サイトのログイン切れ → 該当サイトに手動ログイン → 再検索
  3. Service Worker が固まってる → chrome://extensions/ → サービスワーカーを「停止」 → 自動で再起動
  4. それでもダメ → 拡張アイコン右クリック → 「拡張機能を管理」 → 更新ボタン（🔄）で再読み込み

REINS画像アップロードが遅い・止まる（47秒以上）

  • ログの via=... を確認 → どのホストが効いているか
  • catbox長期障害 → 自動で30分クールダウンしてフォールバック
  • 2週間に1回・朝起こる謎の遅延 → SW Cold Start + DNS遅延の可能性。1回成功すれば治る
  • 全ホストダメ → all_hosts_failed: ... がログに出る

SUUMO入稿が失敗した

  • 確認画面でエラー出た → 手動でエラー修正して登録ボタン押下
  • 登録完了画面に到達すれば Phase6が自動検知 → 60分以内ならGASに通知される
  • 60分超えてしまった or 一度諦めた → AdminPage の「submitting 放置物件」セクションで救済

SUUMO候補物件が submittingで残ってる

  • AdminPage 上部に表示される
  • 「手動で入稿完了済みにする」or「却下扱いで戻す」で対処
  • 30分タイムアウトで自動的に approved に戻る仕組みもある

Chrome拡張の変更が反映されない

  • chrome_extension はローカル読み込み → chrome://extensions/ で更新ボタン押下
  • Service Worker は5分idle で終了 → 次のイベントで新コードがロードされる

GASコードの変更が反映されない

  • gas_deploy.sh 実行で全 deployment を新版に更新
  • nice-feistel ワークツリーが main と同期されている必要あり（gas_deploy.shが警告してくれる）

スプレッドシート

  • 検索条件: 顧客マスタ
  • 承認待ち物件: 自動取得後、承認前に貯まる
  • 通知済み物件: 顧客に通知した物件の room_id 履歴
  • SUUMO巡回条件: 巡回検索の条件
  • SUUMO候補物件: 巡回ヒット物件・承認状態
  • SUUMO掲載管理: 入稿成功した物件・PV/問合/危険度スコア
  • LINE Users: LINE userId ↔ 顧客名 マッピング

その他

  • Discord webhook: 物件通知（forumチャンネル・スレッド形式）
  • 画像ホスティング: catbox→0x0→telegraph→imgbb→freeimage→tmpfiles→pixeldrain→imgur
変更後の運用ルール:
  • GASファイルは nice-feistel ワークツリーでのみ編集 → 他ワークツリーで触ると gas_deploy.sh 時に上書き消滅
  • デプロイ前に git merge main でnice-feistelをmainに同期
  • Chrome拡張の変更は再読み込みだけでOK（デプロイ不要）
